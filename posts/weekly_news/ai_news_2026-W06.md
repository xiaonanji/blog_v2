### Updating our Model Spec with teen protections
- 来源：OpenAI Blog
- 发布：2025-12-18 22:00
- 收录：2026-02-06 00:06
- 链接：https://openai.com/index/updating-model-spec-with-teen-protections

**摘要**
- OpenAI 于 2025-12-18 更新了 Model Spec，新增并明确了面向未满 18 岁用户的 U18 原则，旨在将模型行为与青少年发展的需求相匹配，优先强调预防、透明和早期干预。
- U18 原则针对 13–17 岁青少年设立四项承诺：将青少年安全置于首位、鼓励线下及可信资源支持、以尊重且发展适宜的方式对待青少年、并设定明确的透明预期。
- 在制定过程中，OpenAI 征询并预览了外部专家意见（包括美国心理学会 APA），APA 强调为青少年采取更保护性的做法并配合成人监督的重要性。
- Model Spec 更新还具体说明了模型在高风险话题（如自伤自杀、性化角色扮演、暴力/露骨内容、危险活动与物质、饮食失调、隐瞒不安全行为）上应采取更严格的引导与替代方案，并鼓励获取离线支持或危机资源。
- 除了规范文本，OpenAI 在产品层面扩展了家长控制与保护措施，覆盖群聊、ChatGPT Atlas 浏览器和 Sora 应用，并通过与 ThroughLine 合作在 ChatGPT 与 Sora 中展示本地化求助热线。
- 在技术手段上，OpenAI 正在逐步部署年龄预测模型以在消费者计划中自动应用青少年保护；若年龄信息不确定，将默认 U18 体验并提供成人验证途径。
- OpenAI 表示该 U18 指南将与 GPT‑5.2 的能力、专家委员会（Well‑Being Council）与 Global Physician Network 的临床输入一起被持续迭代，以在产品与模型层面不断加强青少年安全。

**意义**
此更新把“模型行为的规范化”具体化为面向青少年的安全守则与工程化措施，既改变模型在高风险情境中的响应策略，也推动产品与模型协同施行保护，可能成为行业青少年安全实践的参考范式。

**标签**：模型规范 / 青少年安全 / U18原则 / 家长控制 / 年龄预测 / GPT-5.2 / 伦理与对齐

### Reinforcement Learning Heats Up, White House Orders Muscular AI Policy, Computer Use Gains Momentum, Fine Control of Fine-Tuning
- 来源：DeepLearning.AI The Batch
- 发布：2025-01-30 09:48
- 收录：2026-02-05 23:31
- 链接：https://www.deeplearning.ai/the-batch/issue-286/

**摘要**
- DeepSeek 在中国发布了开源权重模型 DeepSeek-R1，其基准表现可与 OpenAI 的 o1 相当，并以 MIT 许可发布，激发市场与舆论关注；消息发布后美股出现短暂抛售（Nvidia 等科技股下挫）。
- 作者指出中国在生成式 AI 上快速追赶美国，例举 Qwen、Kimi、InternVL 与 DeepSeek 等模型，并警示如果美国遏制开源，开源层面的供应链可能被中国主导，进而影响价值观与产业选择。
- 开源权重正在压低基础模型层的使用成本与门槛：作者比较了令牌成本，OpenAI o1 每百万输出令牌约 60 美元，而 DeepSeek-R1 约 2.19 美元，近 30 倍差距显著影响商业模式与应用构建机会。
- 强化学习（RL）再次成为提升 LLM 推理与 chain-of-thought 能力的重要路径：DeepSeek-R1（及 R1-Zero）和 Kimi k1.5 用微调 + 强化学习改进长链路推理；团队还通过先做监督微调再做 RL 来缓解语言混杂等问题。
- 计算资源与成本优化显著：DeepSeek 团队在受美方芯片禁运情况下，用 H800 等受限卡与多项优化把（不含研究成本的）训练算力成本压缩到不足 600 万美元，说明‘放大规模’并非唯一进步路径，但长远算力需求仍可能上升。
- OpenAI 推出能在网页上代替人完成具体操作的 Agent“Operator”，其 Computer-Using Agent（CUA）在 WebVoyager 上成功率 87%，在 OSWorld 上成功率为 38.1%；这一类能调用网页/桌面元素的代理产品正快速进入消费端与企业端。
- 关于微调与合成数据研究，Cohere 提出的“active inheritance”方法通过从多次生成中选择最符合目标特征的合成样本来微调模型，能显著降低模型输出毒性（如 Mixtral 的最大毒性由 65.2 降到 43.2），但在若干基准上对总体性能有时存在提升或下降的权衡。

**意义**
DeepSeek-R1 的开源发布与低成本使用，连同强化学习与合成数据的微调进展，正重新塑造基础模型层的竞争格局与应用生态；与此同时，美国政策方向与大规模算力投资（如 Stargate）将决定技术主导权与产业走向。

**标签**：DeepSeek-R1 / 开源权重 / 强化学习 / 模型微调 / Operator / 美国AI政策 / 算力成本

### How Nano Banana got its name
- 来源：Google Blog
- 发布：2026-01-16 03:06
- 收录：2026-02-05 23:41
- 链接：https://blog.google/products-and-platforms/products/gemini/how-nano-banana-got-its-name/

**摘要**
- 2025年7月底，Google 团队在准备首个对外版本时进行问题修复与评估，技术名称为 Gemini 2.5 Flash Image，但需要为在公开测试平台 LMArena 上使用的版本取一个对外代号。
- LMArena 是一个通过匿名、众包两两对比评估 AI 模型的平台：用户提交提示，收到两个未标识模型的响应并投票，随后平台公布所用模型身份，以获取真实人类偏好信号。
- 由于许多模型在发布前仍处于迭代中，使用代号很重要；产品经理 Naina Raisinghani 在凌晨 2:30 被催促提交代号，她将自己的昵称 Naina Banana 与 Nano（身高与计算机爱好）合并，提出“Nano Banana”。
- 团队在 8 月初以 Nano Banana 身份在 LMArena 上引入该 Flash 模型（技术名 Gemini 2.5 Flash Image），模型展示出显著的编辑能力，包括保持人物相貌一致与将多张图片高质量合成编辑的能力。
- 模型能力与有趣的代号共同触发了社交媒体传播：用户既对模型的性能表示惊讶，又被名字的幽默感吸引，讨论与传播因此快速增长。
- 这段命名与测试过程体现了通过公开、匿名的人类对比测试获取早期性能信号的策略，并说明命名也能成为影响公众认知与传播的因素。
- 文章来源为 Google Blog，发布日期 2026-01-15，文章既回顾了代号的来由，也强调了 LMArena 在模型发布前评估与收集用户反馈方面的角色。

**意义**
该事件表明模型在公开评测平台上的早期人类反馈能推动产品传播与改进，同时代号/命名也会显著影响公众讨论与接受度，对后续发布与营销有直接影响。

**标签**：Gemini 2.5 / Nano Banana / LMArena / 多模态编辑 / 模型评测 / 公开测试 / 社交传播

### Datadog uses Codex for system-level code review
- 来源：OpenAI Blog
- 发布：2026-01-09 11:00
- 收录：2026-02-06 00:02
- 链接：https://openai.com/index/datadog

- 为验证 AI 审查的实际效果，Datadog 构建了基于历史事故的 replay harness，将导致事故的历史 PR 重新跑入 Codex 审查流程，并由当时的工程师判断 Codex 的反馈是否会改变结果。
- 在 replay 测试中，Codex 在被检验的事故样本中识别出超过 10 个案例，约占 22%，这些案例被工程师确认若当时有 Codex 的反馈就可能改变事故走向，显示其能补充人工审查的盲点。
- Datadog 工程师反馈指出 Codex 给出的建议信号高且可读性好，能发现传统静态分析或规则化工具难以捕捉的系统级问题，例如跨模块交互、缺失的跨服务测试覆盖和 API 合约变更带来的下游风险。
- Codex 的审查能力不仅局限于 diff 本身，而是能把 Pull Request 的意图与整个代码库和依赖关系联系起来，甚至执行代码和测试以验证行为，从而把审查从样式/检测转向面向系统风险的判定。
- 在内部推广后，超过 1,000 名工程师常规使用 Codex，相关讨论多在 Slack 等非正式渠道自然流转，团队认为 Codex 帮助他们把人工注意力从低层次检测转向设计与架构审查。
- Datadog 团队将 Codex 视为核心可靠性系统：它能在规模化交付中提升发版信心、暴露跨服务耦合风险并减少潜在事故，而不是单纯替代人工评审的角色。

**意义**
将系统级推理的 AI 审查嵌入日常代码审查流程，Datadog 实现了可量化的事故预防效果与可扩展的高置信审查，推动从速度导向到风险导向的审查范式转变。这为其他依赖复杂分布式系统的企业提供了可借鉴的路径，用 AI 增强可靠性而非仅提升循环效率。

### Powering tax donations with AI powered personalized recommendations
- 来源：OpenAI Blog
- 发布：2026-01-27 11:00
- 收录：2026-02-05 23:51
- 链接：https://openai.com/index/trustbank

**摘要**
- 日本的“ふるさと納税”（hometown tax donation）允许纳税人将部分税款捐给心仪的市町村以支持地方，但因人口向大城市集中导致乡镇税基萎缩，系统通过税收抵免和“返礼品”激励捐赠者。大量市町村与海量返礼品目录使得用户在选择时感到迷茫与操作复杂。
- Furusato Choice（由TRUSTBANK运营）是日本最大的返礼品平台之一，平台上约有760,000件返礼品，界面帮助许多尤其是首次参与者完成捐赠与选择。为解决用户比较与发现难题，TRUSTBANK决定引入 AI 来提供个性化推荐与搜索改进。
- TRUSTBANK 使用 OpenAI API 开发了 Choice AI 功能，并与 Recursive（现为 OpenAI 官方服务伙伴）合作：Recursive 从规划到会话代理实现并构建了 RAG 系统，TRUSTBANK 负责准备返礼品数据库、定义需求并完成应用集成与上线。
- Choice AI 的核心是多代理（multiagent）架构：路由模型解析用户意图并分派给 Search Agent、Recommendation Agent、Greeting Agent 等专门代理；各代理可调用子代理与工具以实现多步协调与更精确的意图响应。个性化通过基于用户信息动态生成 prompt 实现，系统默认运行 GPT‑4.1 系列（默认 GPT‑4.1 mini，按延迟/准确性在测试中动态切换模型规模）。
- 产品层面的改进包括提前展示推荐、扩展推荐品类、优化对模糊请求（如“给父母的礼物”）的响应流程，从而提高发现效率。团队并引入受控随机性以在各都道府县间变换搜索结果，避免集中于少数热门市町村或商品，从而提升地区多样性与公平性。
- 实测数据显示使用 Choice AI 的用户转化率高于仅用站内搜索的用户，原因在于 AI 能从用户难以表达的偏好与预算中抽取信息并给出具体返礼品建议。TRUSTBANK 后续计划将 AI 扩展为端到端的个性化‘礼宾’式服务，进一步提升平台对用户与地方自治体的连接价值。

**意义**
此案例展示了通过多代理 LLM 与产品化工程将大规模商品目录的发现问题转化为可量化的商业与社会效益（提高转化率、扶持小众地区）。它也是企业借助外部 AI 服务伙伴快速落地复杂推荐系统的参考范例。

- 架构原则一：为现实复杂性而建——单次企业请求通常跨预订引擎、忠诚度数据库、支付与政策逻辑等多系统，Netomi 通过治理的编排管线维持状态与一致性。

- 在高度监管行业（如牙科保险和医疗保险场景）中，Netomi 每年处理近 200 万次提供者请求，展示了在合规与审计要求下保持确定性与可追溯性的能力。
- Netomi 的蓝图强调：把模型作为推理骨干，而通过系统工程确保智能在企业级环境中可操作、可审计并具备高可用性与可伸缩性。

**意义**
对于希望将 agentic AI 推向企业生产环境的公司，必须同时解决多步规划、低延迟并发执行与内置治理，单靠模型能力不足以满足复杂业务与合规需求。Netomi 的实践提供了一个可复制的工程蓝图，降低企业部署风险并提升可用性。

- 描述了从用户输入到模型采样（inference）的完整流程：提示构建（包含 role 优先级 system/developer/user/assistant）、token 化、增量生成与流式输出，以及模型可能发起的工具调用（function call）和代理执行后将结果回写到 prompt 的机制。
- 详述 Codex 与 Responses API 的交互实现：客户端构建 JSON input 列表并通过 HTTP POST 发起请求，服务器将 JSON 转换为模型 prompt 并通过 SSE 流返回事件（如 response.output_text.delta、response.output_item.added 等），Codex 将这些事件转化为后续输入或 UI 流式展示。
- 强调性能与效率实践：提示缓存（prompt caching）通过保证新请求为旧 prompt 的精确前缀实现缓存命中，将模型采样成本从二次成长降为线性；为提高命中率，建议把静态指令/示例置于 prompt 前端，把变量内容放后端。
- 讨论上下文窗口与 compaction 策略：随着会话增长 prompt 会膨胀并可能耗尽上下文窗口，Codex 通过在达到阈值后用更小的、代表性条目替换 input（自动或/和手动 compact）来回收上下文位宽。
- 揭示若干设计权衡：Codex 当前不使用 previous_response_id 保持每次请求无状态以支持 Zero Data Retention（ZDR）配置，这简化提供方实现且兼顾不保留数据的用户需求，但会增大网络 JSON 总量（可由缓存与采样成本比重缓解）。
- 列举会导致缓存未命中或失效的操作类型：中途更改可用工具、更换目标模型、更改 sandbox/approval mode 或工作目录等，并说明通过追加新消息体现配置变更以尽量减少对缓存的破坏。
- 文章同时给出工程实践与最佳示例（如将工具与图像保持一致以便缓存命中）并说明后续文章将深入 CLI 架构、工具使用实现与沙箱模型等细节。

### Unlocking the Codex harness: how we built the App Server
- 来源：OpenAI Blog
- 发布：2026-02-05 00:00
- 收录：2026-02-05 23:45
- 链接：https://openai.com/index/unlocking-the-codex-harness

**摘要**
- 本文介绍了 Codex App Server 的设计与实现背景，最初作为可复用的 Codex harness，用于在终端 TUI 与 VS Code 等 IDE 间共享同一代理循环，随着内外部采用扩展为稳定的集成协议。
- App Server 是一个长期进程与 JSON-RPC 协议的组合，使用双向 JSON-RPC over stdio（JSONL）作为客户端与服务器之间的通信通道，支持流式事件、审批请求与服务器发起的请求。
- 内部架构包括 stdio reader、Codex message processor、thread manager 与 core threads，thread manager 为每个会话线程启动一个 core session，消息处理器将低层事件转换为稳定的 UI 就绪通知。
- 提出了三类会话原语来表达复杂交互：Item（有 lifecycle：started/delta/completed）、Turn（由一次用户输入触发的一次代理工作）和 Thread（持久会话容器并支持 fork/恢复/归档与持久化历史）。
- 对不同接入面（本地应用/IDE、Codex Web runtime、TUI）给出了实现模式与权衡：本地进程绑定二进制、Web 在容器中运行 App Server 并通过 SSE/HTTP 转发事件、TUI 计划重构为标准客户端模式。
- 讨论了其他集成方式的适用场景与局限：MCP 适合已有 MCP 流程但功能受限，轻量 CLI 与 TypeScript 库适合非交互或内嵌场景；推荐在需要完整 harness 功能时选用 App Server。
- 强调向后兼容与易集成的设计目标：通过稳定的 JSON-RPC 面向多语言绑定（Go/Python/TypeScript/Swift/Kotlin）与可生成的 schema，合作方可在不频繁更新客户端的情况下采用新二进制。

**意义**
Codex App Server 将复杂的多步代理逻辑以稳定、可流式渲染的协议暴露给多种客户端，降低在 IDE、桌面与 Web 中实现一致代理体验的工程成本，推动开发者在真实产品中更容易落地自动化代理能力。

**标签**：Codex / App Server / JSON-RPC / 代理协议 / IDE 集成 / 流式事件

### Scaling PostgreSQL to power 800 million ChatGPT users
- 来源：OpenAI Blog
- 发布：2026-01-22 23:00
- 收录：2026-02-05 23:52
- 链接：https://openai.com/index/scaling-postgresql

**摘要**
- 本文披露了 OpenAI 为支撑约 8 亿 ChatGPT 用户而对 PostgreSQL 进行的大规模扩展工程，目标是将数据库承载能力提升到每秒数百万查询级别，以应对海量在线请求。
- 为扩展读写吞吐，团队广泛使用数据库副本（replicas）来分散读流量并提高可用性，通过读写分离和多副本部署缓解主库压力。
- 引入多层缓存以减少对后端 PostgreSQL 的直接访问，从而显著降低延迟和查询负载，缓存策略成为整体系​​统性能优化的关键一环。
- 通过速率限制（rate limiting）保护数据库免受突发流量和恶意请求冲击，确保流量平稳并为不同客户或服务提供公平配额。
- 实施工作负载隔离，把延迟敏感的在线服务与批处理、分析或后台任务隔离开来，防止“噪声邻居”效应导致生产查询性能退化。
- 文章讨论了在保持关系型数据库一致性与可靠性前提下的工程权衡，包括可用性、延迟、成本与复杂度的取舍，表明传统关系型系统仍可通过架构优化服务大规模 AI 产品。
- 对运营与运维层面的关注包括容量规划、监控告警与故障切换机制，这些措施共同支撑了 Postgres 在高并发环境下的稳定运行。

**意义**
这表明通过副本、缓存、速率限制和工作负载隔离等工程手段，传统关系型数据库 PostgreSQL 可以在大规模在线 AI 服务中达到百万级 QPS 的实用可用性，对大型产品架构选择具有重要参考价值。

- 作者构建了一套评估框架与测试套件，包含13项评估、共24个环境，并将评估分为三类原型：干预（intervention）、过程（process）和输出属性（outcome-property）。
- 实验比较了多款前沿推理模型（例如 Claude 3.7 Sonnet、DeepSeek R1-0528-Qwen3-8B、Kimi K2 Thinking 以及 OpenAI 系列模型），发现大多数前沿推理模型总体上是“相当可监测”的，但并非完美可监测。
- 关键发现包括：监测链式思考比仅监测行动与最终输出效果更好；随着模型产生更长的思路链、或提高推理努力，监测性普遍提升；当前规模的强化学习（RL）训练并未显著降低监测性，且在初期步骤甚至提高了监测性。
- 研究揭示了一个权衡：以较低推理努力运行的大模型，不如用较小模型并提高推理努力在可监测性上更有优势（但会付出更多推理算力，即“可监测性税”）。
- 通过在对话后询问后续问题并监控随后的 CoT，可以在事后提高可监测性，尤其在初始推理努力较低时更为明显；仅看后续答案而不看后续 CoT 改进有限。
- 论文/博客同时指出局限性：每个模型只训练一次、实验多为内分布任务（数学、科学、代码）且评估可能不完全泛化，作者呼吁研究界继续跟踪并努力保存链式思考的可监测性。

**意义**
链式思考监控被视为有望成为可扩展控制的关键机制之一，研究表明保持并测量可监测性对安全部署高风险推理模型具有直接影响。为此需要在模型训练与评估流程中优先保留或强化可监测性，并建立长期追踪指标。

### Keeping your data safe when an AI agent clicks a link
- 来源：OpenAI Blog
- 发布：2026-01-28 11:00
- 收录：2026-02-05 23:49
- 链接：https://openai.com/index/ai-agent-link-safety

**摘要**
- AI 代理在自动打开网页或加载资源时可能将包含敏感信息的完整 URL 发送给第三方，攻击者可通过在 URL 中嵌入如邮箱、文档标题等私有数据实现数据外泄；即便模型不在对话中直接输出敏感内容，后台静默的请求也能泄露信息。
- 攻击手法常结合提示注入：网页内容可以尝试覆盖模型指令或诱导模型发起带有私有参数的请求，且这些请求可能以嵌入图片或链接预览的方式在用户不知情下发生，增加检测难度。
- 仅依赖“可信域名白名单”不足以防御此类攻击：合法站点的重定向可被滥用将请求引至攻击者控制的目的地，而且过于严格的白名单会带来频繁的误报与糟糕的用户体验，促使用户盲目绕过警示。
- OpenAI 提出的核心策略是：仅对已被独立爬虫索引并公开出现过的具体 URL 允许自动抓取；在请求前把待抓取的 URL 和独立索引比对，若未匹配则将其视为未验证并要求用户确认或拒绝自动加载。
- 当链接未通过公开性验证时，产品会向用户展示警示（例如提示链接可能包含会话相关信息），以保持用户控制权并建议用户请求替代来源或摘要，从而降低“静默泄露”的风险。
- 该措施专注于防止通过 URL 本身的特定数据外泄，不能替代对网页内容可信度、社会工程或提示注入的其他防护，因此被作为多层防御（模型级缓解、产品控件、持续红队测试等）的一部分持续迭代。

**意义**
这套基于独立网页索引的验证机制可显著降低 AI 代理在后台静默请求时通过 URL 泄露用户专有数据的风险，从而提升代理在自动化任务中的安全性，但仍需与其他防护层协同以应对更广泛的攻击面。

**标签**：AI代理 / URL泄露 / 提示注入 / 独立网页索引 / 自动抓取 / 防护策略 / 数据泄露
